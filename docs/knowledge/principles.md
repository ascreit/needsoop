# 成功パターン・基本原則

## 基本原則

```
【失敗の90%】開発してから売ろうとした
【成功パターン】売れることを確認してから開発した

フロー: アイデア → 検証（複数方法） → 有料コミット確保 → 開発開始
                        ↓ 失敗
                   ピボット or 撤退（2週間ルール）
```

## お金を払う理由（収益化の3基準）

成功事例に共通する条件。**3つすべて満たす必要がある**：

| 基準 | 質問 | NG例 |
|-----|------|-----|
| **継続利用** | 毎日/毎週使うか？ | 利用規約生成（1回きり） |
| **収益直結** | 時間節約 or 売上増に繋がるか？ | あったら便利程度 |
| **緊急性** | 今すぐ解決したいか？ | いつか対応すればいい |

---

## 失敗パターン（避けるべき）

| パターン | 問題 | 対策 |
|---------|------|------|
| 「欲しい」を直接探す | 既にある | 不便→連想→サービス化 |
| 競合調査なし | 無料ツール多数 | 必ず料金・不満を調査 |
| 需要なし | 作ってから売ろうとした | 検証ファースト |
| 機能過多 | 開発に時間かかりすぎ | MVP1機能のみ |

---

## 注意事項

- **競合調査は必須**（無料で十分なツールがあれば撤退）
- 「トレンドだから」は理由にならない（差別化必須）
- 技術的に面白いかは関係ない（売れるかどうか）
- 2週間検証して反応なければ撤退
- 1アイデアに固執しない（成功者は平均4-5個失敗）
